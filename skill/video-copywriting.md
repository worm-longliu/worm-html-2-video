# 视频文案与字幕规范

## 配音文案格式（voiceover_text.txt）

> **脚本驱动流程说明**：`voiceover_text.txt` 现在是 `script.json` 的派生物
> （由 `npx worm-html-2-video script vo` 生成），其中的时间标记为估算值，
> 仅供人工阅读。权威数据源是 `script.json` 的 `voiceover` / `subtitle` 字段；
> 每个场景的真实时长由 `npx worm-html-2-video voiceover` 测量并写入 `scene_timings.json`，
> 再由 `npx worm-html-2-video sync` 回填到各 `scenes/scene-N.html` 的 `data-duration`。
> 因此文案创作时**无需精确估算时间**，只需控制字数与语速匹配即可。

### 标准格式

```
[场景1: 0-4s]
为了不看 AI 写代码，我写了三个方案。

[场景2: 4-10s]
AI 在写代码，你在疯狂切屏。
Agent 开发卡住不干活也不知道，半天的时间就浪费了。

[场景3: 10-17s]
方案一：HTTP 接口通知。
很多 AI 工具不支持 hook，根本不触发回调机制。
```

### 格式规则

| 元素 | 格式 | 说明 |
|------|------|------|
| 时间标记 | `[场景N: Xs-Ys]` | N=场景编号，X=开始秒，Y=结束秒 |
| 句子分隔 | 换行 | 每个独立句子单独一行 |
| 场景分隔 | 空行 | 场景之间空一行 |
| 标点 | 中文标点 | 句号、逗号、冒号等 |

### 时间标记与场景 `data-duration` 的关系（派生值，仅供参考）

`voiceover_text.txt` 的时间标记是 `script_tool.py` 按语速估算生成的，
`data-duration` 的真实值由 `sync_html.py` 根据 `scene_timings.json` 写入：

```
script.json: {"voiceover": "..."}        ← 只写文案，不写时间
voiceover.py → scene_timings.json: {"duration": 6.2}  ← 真实配音时长
sync_html.py → scenes/scene-N.html: data-duration="6.2"  ← 据时长回填
```

**核心约束**：`data-duration` = 该场景配音真实时长（末场景加 tail buffer），
由工具链自动保证，无需人工对齐时间标记。

## 文案创作规范

### 语速控制

| 内容类型 | 语速 | 字符/秒 | 示例 |
|----------|------|---------|------|
| 正常描述 | 标准 | 4-5 字/秒 | "放在桌面角落，一眼就能看到" |
| 技术术语 | 稍慢 | 3-4 字/秒 | "Rust 加 Tauri v2" |
| 情绪表达 | 稍快 | 5-6 字/秒 | "疯狂切屏！半天浪费了！" |
| 数字/英文 | 慢 | 2-3 词/秒 | "GitHub，worm-longliu" |

### 字数与时长对照表

| 场景时长 | 建议总字数 | 句子数 | 备注 |
|----------|-----------|--------|------|
| 4s | 14-18 字 | 1-2 句 | 开场/结尾 |
| 6s | 22-28 字 | 2-3 句 | 痛点描述 |
| 7s | 26-33 字 | 2-3 句 | 方案展示 |
| 8s | 30-38 字 | 3-4 句 | 产品演示 |

### 计算公式

```
预估时长 = 中文字数 ÷ 4.5 + 英文单词数 × 0.5 + 停顿数 × 0.3
```

### 语气风格矩阵

| 场景类型 | 语气 | 关键词 | 示例 |
|----------|------|--------|------|
| 开场 | 吸引注意/制造悬念 | 为了、你有没有、真的 | "为了不看 AI 写代码" |
| 痛点 | 共鸣吐槽 | 疯狂、浪费、卡住 | "半天的时间就浪费了" |
| 失败方案 | 否定/遗憾 | 不支持、不行、暴涨 | "CPU 占用暴涨" |
| 成功方案 | 兴奋/肯定 | 完美、实时、不占 | "不占资源、实时响应" |
| 产品展示 | 自豪/推荐 | 最终、小巧、即用 | "3MB 小灯，运行即用" |
| 结尾 | 引导/期待 | 关注、下集、求Star | "关注我，下集见！" |

### 文案质量检查清单

- [ ] 每句话独立完整，不依赖上下文
- [ ] 避免书面语，使用口语化表达
- [ ] 技术内容通俗化，外行也能听懂大意
- [ ] 场景文字总量 ≤ 场景时长 × 5 字
- [ ] 英文/数字处有合理停顿
- [ ] 前后逻辑通顺，起承转合清晰

## 字幕规范（内嵌于 HTML）

字幕不再单独生成 SRT 文件,而是直接写入各 `scenes/scene-N.html` 的 `SUBTITLES` 数组。
HTML 中预留字幕条 DOM,渲染循环按当前时间切换文本,截图时随帧捕获。

### SUBTITLES 数据格式

```javascript
const SUBTITLES = [
  { start: 0.3, end: 3.7, text: "为了不看 AI 写代码\n我写了三个方案" },
  { start: 4.2, end: 7.1, text: "AI 在写代码，你在疯狂切屏" },
];
```

- `start` / `end` 单位为秒,在全局时间轴上
- `text` 字符串中 `\n` 触发换行
- 数组可为空 (无字幕视频)

### 字幕时间轴原则

> 新机制（默认）：`voiceover.py` 按句拆分配音并测每句真实时长，写入
> `scene_timings.json` 的 `segments`；`sync_html.py` 据此生成字幕——文本=配音
> 原文，时间=该句真实起止。下方原则描述的是无 `segments` 时的回退算法。

1. **场景首留白**: 字幕起始 = 场景开始 + 0.3s (等画面稳定)
2. **场景尾留白**: 字幕结束 = 场景结束 - 0.3s (避免切换时残留)
3. **句间间隔**: 两条字幕间隔 ≥ 0.2s (避免视觉混淆)
4. **最短时长**: 单条字幕 ≥ 1.5s (确保读者看清)
5. **最长时长**: 单条字幕 ≤ 5.0s (回退算法上限；segments 模式下随句子真实时长)

### 字幕换行规则

| 规则 | 说明 | 示例 |
|------|------|------|
| 每行最多 20 字符 | 超出则 `\n` 换行 | `Agent开发卡住不干活\n也不知道` |
| 优先标点处断行 | 逗号、句号、顿号 | `不占资源、\n实时响应` |
| 最多 2 行 | 超过则拆分为多条字幕 | — |
| 英文不拆词 | 整词保持在同一行 | `Rust + Tauri v2` 不换行 |

### 字幕文本与配音文案的关系

| 文件 | 内容 | 时间标记 | 用途 |
|------|------|----------|------|
| `voiceover_text.txt` | 配音文案 | `[场景N: Xs-Ys]` | TTS 输入 |
| `scenes/scene-N.html` SUBTITLES | 显示文本 | `{ start, end, text }` | 视频显示 |
| `subtitle.srt` (旧) | 显示文本 | `HH:MM:SS,mmm` | 第三方平台上传 (可选) |

新机制下,`SUBTITLES[i].text` 与配音原文按句完全一致 (由 `segments` 直接生成,
无需手动同步)。仅当采用回退算法 (无 `segments`) 时,才需人工保证字幕文本与配音一致。

> **多音字提示**: Edge-TTS 纯文本无法可靠强制多音字读音 (如"一行"可能读
> yixing)。`voiceover.py` 的 SSML `<phoneme>` 方案实测会令该场景时长异常暴增,
> 不可用。规避方法:在 `script.json` 的 `voiceover` 中改用无多音字歧义的等价
> 表述 (如"一行命令"→"一条命令")。

```css
.subtitle-bar {

```

### 字幕与配音对齐验证

新机制下字幕由 `segments` 直接生成，文本与时间天然一致，通常无需验证。若需排查，
检查 `scene_timings.json` 中每场景的 `segments`：每条 `text` 应是 `voiceover` 的一个
完整句子，`start`/`end` 应在该场景 `[0, duration]` 范围内连续无重叠。

### 何时需要重跑 sync（而非手动调字幕）

字幕由 `sync` 自动生成，**不要手动改 `SUBTITLES`**（会被下次 sync 覆盖）。以下情况
重跑 `voiceover` → `sync`：

- 修改了 `script.json` 的 `voiceover`（字幕文本随配音原文变化）
- 重新生成配音后时长变化（字幕时间随真实时长变化）
- 想调整某句断句：在 `voiceover` 中用句末标点（`。！？`）控制拆分点

> 关键名词/数字的颜色突出仍可在 `scene-N.html` 的字幕 DOM 外另行叠加样式，
> 但 `SUBTITLES` 数组本身由 sync 维护。

## 封面文字设计

### 排版规范

| 元素 | 字号 | 颜色 | 位置 |
|------|------|------|------|
| 主标题 | 72px | #ffffff | Y: 650-800px |
| 副标题 | 36px | #e6e6e6 | 主标题下方 60px |
| 标签 | 28px | #00ff88 | 副标题下方 40px |

- **顶部 180px** — 不放文字（状态栏/刘海屏遮挡）


## 分镜脚本模板（script.md）

### 标准格式

```markdown
# 分镜脚本 - [视频主题]

## 基本信息
- **总时长：** XXs
- **场景数：** N 个
- **目标平台：** 抖音/视频号

## 场景1: [场景标题]（0-4s）
- **画面：** [视觉描述]
- **配音：** "[完整配音文案]"
- **动画：** [动画描述，用 → 连接顺序]
- **关键元素：** [列出核心UI组件]

## 场景2: [场景标题]（4-10s）
...
```

### 分镜脚本创作流程

```
1. 确定总时长和场景数量
2. 分配每个场景的时间预算
3. 编写配音文案（按语速验证字数）
4. 设计画面和动画（确保与配音节奏匹配）
5. 验证：场景时长 ≥ 配音时长 + 0.5s 余量
```

## 录音文案优化技巧

### 增强流畅性

| 问题 | 优化前 | 优化后 |
|------|--------|--------|
| 书面语 | "该方案存在以下问题" | "这方案有个大问题" |
| 长句 | "通过系统文件事件机制实现" | "用系统事件，一行代码搞定" |
| 被动语态 | "CPU 被大量占用" | "CPU 直接拉满" |
| 缺乏停顿 | "方案一HTTP接口通知" | "方案一，HTTP 接口通知" |

### Edge-TTS 发音优化

| 场景 | 技巧 | 示例 |
|------|------|------|
| 英文缩写 | 字母间加空格 | `A I` → 念 "A-I" |
| 数字 | 加单位说明 | `3MB` → "三兆" |
| 停顿 | 加逗号/句号 | "方案一，HTTP" |
| 语气强调 | 加感叹号 | "求 Star！" |
| 连读问题 | 加空格断开 | `v2` → "v 2" |

### 配音文案自检

```
□ 大声朗读一遍，是否自然？
□ 是否有拗口的地方？
□ 英文/数字是否需要添加停顿？
□ 每句话是否都有明确的信息传达？
□ 情绪变化是否自然？（痛点→方案→成果）
□ 总字数是否匹配总时长？
```
